GitHub Copilot: I’ll explain how Step 2 (API endpoints) will work and the end-to-end flow from form submit to the persisted JSON.

**Hoe Step 2 werkt**
- Voeg twee API-routes toe aan server.js:
  - `GET /api/proposals` → leest en retourneert de array `proposals` uit `src/data/stagevoorstellen.json`.
  - `POST /api/proposals` → accepteert een JSON-body, voegt het voorstel toe aan de array en schrijft het bestand terug.
- De server gebruikt node `fs` (promises) om het JSON-bestand te lezen en te overschrijven. Het bestand `src/data/stagevoorstellen.json` is de enige bron (single mock JSON).

**Dataflow — stap voor stap (indienen → permanent mock JSON)**
1. Gebruiker vult het formulier in en klikt `Indienen` (in `src/student/formulier.js`).
2. Client valideert velden en bouwt een voorstel-object:
   - voorbeeldvelden: `id`, `studentNaam`, `studentNummer`, `bedrijfNaam`, `bedrijfAdres`, `mentorNaam`, `mentorEmail`, `opdrachtOmschrijving`, `periodeStart`, `periodeEind`, `status: 'wachten'`, `ingediendOp` (ISO-timestamp).
3. Client roept `dataService.saveProposal(proposal)` aan.
   - `dataService` probeert een `POST /api/proposals` met `Content-Type: application/json`.
   - Als de server succesvol reageert (201), is het voorstel op de server geschreven naar `src/data/stagevoorstellen.json`.
   - Als de server niet bereikbaar is, valt `dataService` terug op `localStorage` (zodat het formulier nog steeds lokaal werkt).
4. Na succesvolle save redirect de client naar de wachten-pagina (`/?role=wachten`).
5. De wachten-pagina (of `dataService.getLatestProposal()`) haalt gegevens op:
   - eerst via `GET /api/proposals` (server) en toont het meest recente voorstel;
   - als server niet beschikbaar: leest uit `localStorage` of uit de oorspronkelijke mock (fallback).
6. Het JSON-bestand `src/data/stagevoorstellen.json` bevat voortaan alle ingediende voorstellen en blijft op disk staan (permanent voor deze mock backend).

- Maakt persistente mock-backend mogelijk zonder echte database.

- Toont dezelfde data aan alle gebruikers wanneer server actief is (shared state).

- Schrijven is read-modify-write; er kan race zijn bij intensieve gelijktijdige requests — acceptabel voor mock, maar voor productie gebruik een echte DB or file-locking.

- Als je ontwikkelt met Vite/dev-server moet je ofwel de Express-server gebruiken (`npm run server`) of een proxy instellen zodat fetch-requests naar `/api/proposals` werken.
